Project OverviewLeadCode is a competitive coding platform — similar to LeetCode — with a critical differentiator: every accepted submission is automatically committed to the user's own GitHub repository and optionally shared as a LinkedIn activity post.What makes LeadCode differentUsers connect their GitHub and LinkedIn accounts once during onboarding.When any solution is accepted, LeadCode auto-pushes the code to their GitHub repo, organized by difficulty and problem name.Users can opt-in to auto-post their accepted solutions to LinkedIn, building a public coding portfolio passively.Every submission is recorded — giving users a visible commit history that functions as a live resume artifact.Core User FlowUser signs up and connects GitHub (required) and LinkedIn (optional).User solves a problem in the Monaco code editor.On "Submit", code is evaluated by the Judge0 engine against hidden test cases.If verdict = Accepted: a background job fires to push code to GitHub + optionally post to LinkedIn.User sees their accepted solutions in a profile page and via their GitHub commit history.Phase 1 — Authentication & User IdentityThis is the foundation. Everything else depends on a stable user identity with securely stored third-party OAuth tokens.Step 1.1 — Core Auth SetupUse Supabase Auth for email/password and magic link sign-in. Supabase handles JWT issuance, session refresh, and RLS (row-level security) on your Postgres database out of the box.Database schema for the users table:users: id (uuid), username, email, avatar_url, created_at, github_connected (bool), linkedin_connected (bool)STEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION1.1 — Email AuthSupabase magic links expire after 1 hour — users on slow networks may miss them and get frustrated.Enable both magic link AND password auth so users can choose. Set magic link expiry to 24h in Supabase dashboard under Auth > Email settings.1.2 — Session PersistenceSupabase sessions expire by default after 1 hour, forcing users to re-login mid-solve.Enable "Persist sessions" in Supabase. Set JWT expiry to 7 days. Use the Supabase JS client's onAuthStateChange listener to silently refresh tokens.Step 1.2 — GitHub OAuth IntegrationGitHub OAuth is mandatory — it is the core feature of LeadCode. Users without GitHub connected cannot have their solutions saved automatically.OAuth Scopes required:repo — needed to create files and commits in the user's repositoryuser:email — needed to identify the user and link accountsFull OAuth Flow:Redirect user to: https://github.com/login/oauth/authorize?client_id=YOUR_ID&scope=repo,user:email&state=RANDOM_STATEGitHub redirects back to your callback URL with ?code=XXXServer-side: POST to https://github.com/login/oauth/access_token with code + client_secretParse the returned access_token from the response bodyEncrypt and store the token in your DB against user_idCall GET /user on the GitHub API to fetch their github_usernamePrompt the user to select or create a GitHub repo for their solutions// Exchange code for token (server-side only — never expose client_secret)const res = await fetch("https://github.com/login/oauth/access_token", {  method: "POST",  headers: { Accept: "application/json" },  body: JSON.stringify({ client_id, client_secret, code })});const { access_token } = await res.json();// Store encrypted: await encrypt(access_token) before saving to DBSTEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION1.2a — Scope Too BroadThe "repo" scope gives access to ALL user repos. Users may feel this is invasive and refuse to connect.Clearly explain in the OAuth consent UI: "We only write to the specific repo you choose." Consider requesting the "public_repo" scope if users primarily want a public solutions repo — it is less invasive.1.2b — Token EncryptionStoring plain GitHub access tokens in DB is a critical security risk. A DB breach exposes all user repos.Encrypt tokens at rest using AES-256-GCM before storing. Use a server-side KMS key (AWS KMS or GCP KMS). Decrypt only in the background job worker when needed to make API calls.1.2c — Token RevocationUser revokes the GitHub app manually. All subsequent pushes will fail silently with 401 errors.Catch 401 responses from GitHub API in your worker. Mark github_connected = false and send the user an in-app notification: "GitHub connection expired — reconnect to resume auto-saves."1.2d — Repo SelectionUsers may not know which repo to link, or may accidentally pick a wrong one containing other code.After OAuth, show a modal listing their existing repos (via GET /user/repos API). Also offer: "Create new repo called 'leadcode-solutions'" as the top option with a single click.Step 1.3 — LinkedIn OAuth IntegrationLinkedIn uses OAuth 2.0 with OpenID Connect. The w_member_social scope allows posting on behalf of the user. This requires LinkedIn to review and approve your app.OAuth Scopes required:openid — identity verificationprofile — access to name and profile picturew_member_social — permission to create posts on behalf of the user// LinkedIn OAuth URLhttps://www.linkedin.com/oauth/v2/authorization  ?response_type=code  &client_id=YOUR_CLIENT_ID  &redirect_uri=https://leadcode.app/auth/linkedin/callback  &scope=openid%20profile%20w_member_social  &state=RANDOM_STATESTEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION1.3a — App Review RequiredLinkedIn requires manual app review before granting w_member_social to other users. Can take 1-4 weeks.Apply for app review immediately at developer.linkedin.com. In the meantime, launch with GitHub-only. LinkedIn posting can be a v2 feature. Use your own account for demos.1.3b — Token ExpiryLinkedIn access tokens expire after 60 days. Refresh tokens also expire eventually. This breaks auto-posts silently.Store token_expiry timestamp alongside the access_token. Before each post, check if expiry < now + 7 days. If so, attempt refresh. If refresh fails, mark linkedin_connected = false and notify the user.1.3c — Post SpamIf a user solves 10 problems in one day, they get 10 LinkedIn posts. This is extremely annoying for their network.Implement a daily digest option: batch all that day's accepted solutions into a single post at 8 PM. Also add a per-submission toggle: "Share this on LinkedIn" checkbox visible before submitting.Phase 2 — Problem System & Code Execution EngineThe problem editor and code runner are the core product experience. Users need a fast, reliable in-browser coding environment with accurate verdict feedback.Step 2.1 — Problem Database Schemaproblems:  id           UUID primary key  title        TEXT          -- e.g. "Two Sum"  slug         TEXT unique   -- e.g. "two-sum" (used in GitHub folder path)  difficulty   ENUM(easy, medium, hard)  tags         TEXT[]        -- e.g. ["array", "hash-table"]  description  TEXT          -- HTML or Markdown  starter_code JSONB         -- { python: "...", js: "...", java: "..." }  test_cases   JSONB         -- [{ input: "...", expected_output: "..." }]  hidden_tests JSONB         -- same format, not exposed to usersStep 2.2 — Monaco Code Editor (Frontend)Monaco Editor is the same editor that powers VS Code. It provides syntax highlighting, IntelliSense, and keyboard shortcuts that developers already know.npm install @monaco-editor/react// Usage in your React component:import Editor from "@monaco-editor/react";<Editor  height="500px"  language={selectedLanguage}   // "python", "javascript", etc.  value={code}  onChange={(val) => setCode(val)}  theme="vs-dark"/>STEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION2.2 — Editor Load TimeMonaco Editor is a large bundle (~2MB). First load is slow on mobile connections.Use dynamic import with Next.js: const Editor = dynamic(() => import("@monaco-editor/react"), { ssr: false }). Monaco loads only when the problem page is visited, not on the homepage.2.2 — Language SwitchingWhen user switches language mid-solve, their written code gets wiped, causing frustration.Store code per-language in a state object: { python: "...", javascript: "..." }. When switching, save current code to that language key. Restore when switching back. Persist in localStorage as a draft.Step 2.3 — Judge0 Code Execution EngineJudge0 is an open-source code execution system that runs submitted code in isolated Docker containers. It supports 60+ programming languages and can be self-hosted or used via their cloud API.Self-hosting Judge0 (recommended for production):Deploy Judge0 on a separate server (Railway or Fly.io) using their Docker Compose setup.Judge0 uses RabbitMQ for job queuing and Redis for caching — both are included in the compose file.Point your Next.js API route to your Judge0 instance URL.Never expose Judge0 directly to the browser — always proxy through your own API.// Submit code to Judge0 (from your API route, not client)const submission = await fetch("https://your-judge0.railway.app/submissions", {  method: "POST",  headers: { "Content-Type": "application/json" },  body: JSON.stringify({    language_id: 71,          // 71 = Python 3, 63 = JavaScript    source_code: btoa(code), // base64 encode    stdin: btoa(testInput),    expected_output: btoa(expectedOutput)  })});const { token } = await submission.json();// Poll GET /submissions/{token} until status is not "Processing"STEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION2.3a — Polling LatencyPolling Judge0 every second for results adds unnecessary load and feels slow to the user.Use exponential backoff polling: check at 500ms, 1s, 2s, 4s. Show a real-time spinner with elapsed time counter so users know the system is working. Most submissions return in under 2 seconds.2.3b — Cheating via stdinUsers can print the expected_output directly using sys.stdin tricks to fake acceptance.Never send expected_output to Judge0 from the client. Run all hidden tests server-side. Compare outputs server-side. The client only ever sees "Accepted" or "Wrong Answer" — never the expected output for hidden tests.2.3c — Resource AbuseA user can submit infinite loops or memory-exhausting code, crashing your Judge0 instance.Set Judge0 limits: cpu_time_limit=5 (seconds), memory_limit=131072 (128MB). Rate-limit submissions per user: max 10 per minute. Add a run queue so bursts don't crash the server.2.3d — Language CoverageAdding support for a new language requires updating your problem starter code templates, test runners, and file extension mappings.Create a languages config file: { id: 71, name: "Python 3", ext: "py", judge0_id: 71 }. All language-specific logic references this config. Adding a new language means adding one entry to this file.Phase 3 — GitHub Auto-Push on Accepted SubmissionThis is the defining feature of LeadCode. When a submission is accepted, the code is committed to the user's GitHub repo automatically — no manual copy-paste required.Step 3.1 — Repository Folder StructureSolutions are organized in a consistent folder structure so the GitHub repo stays clean and browsable:leadcode-solutions/  README.md                        ← auto-generated profile page  solutions/    easy/      two-sum/        solution.py        README.md                    ← problem description + your stats    medium/      longest-substring/        solution.js        README.md    hard/      median-of-two-sorted-arrays/        solution.java        README.mdStep 3.2 — GitHub Contents API (No Git CLI Needed)You do NOT need to run git commands on your server. GitHub's REST API has a Contents endpoint that creates and updates files directly:// Create or update a file via GitHub APIconst path = `solutions/${difficulty}/${slug}/solution.${ext}`;const url = `https://api.github.com/repos/${username}/${repo}/contents/${path}`;// First check if file exists to get its SHA (required for updates)const existing = await fetch(url, {  headers: { Authorization: `Bearer ${githubToken}` }}).then(r => r.ok ? r.json() : null);// Then create or updateawait fetch(url, {  method: "PUT",  headers: {    Authorization: `Bearer ${githubToken}`,    "Content-Type": "application/json"  },  body: JSON.stringify({    message: `✅ Solved: ${title} [${difficulty}] — LeadCode`,    content: Buffer.from(code).toString("base64"),    sha: existing?.sha  // omit if creating new file  })});STEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION3.2a — Commit AuthorThe commit will show as authored by your app's GitHub bot account, not the user. Their GitHub contribution graph won't update.In the Contents API request, add an "author" field: { name: userGithubUsername, email: userEmail }. GitHub will attribute the commit to the user, and it will count toward their contribution graph.3.2b — File Already ExistsIf a user solves the same problem again (to improve their solution), the PUT will fail without a SHA.Always do a GET before PUT. If GET returns 200, extract the sha field from the response. Pass sha in the PUT body. If GET returns 404, omit sha — it's a new file. Handle both cases in a helper function.3.2c — Auto-generated READMERaw code files in GitHub are hard to browse. There is no context about what the problem was.Auto-generate a README.md alongside each solution file containing: problem title, difficulty, link to LeadCode problem, submission date, runtime and memory stats. Push both files in separate API calls (GitHub Contents API does not support batch commits).3.2d — Rate LimitsGitHub API allows 5000 requests/hour per token. A power user solving many problems fast could hit this.Track API usage per user token. GitHub returns rate limit headers (X-RateLimit-Remaining). If remaining < 50, queue the push for the next hour instead of failing. Show user: "GitHub push queued — will appear shortly."Phase 4 — LinkedIn Auto-Post on Accepted SubmissionLinkedIn posting is opt-in and controlled entirely by the user. The goal is to help users build a public presence without spamming their connections.Step 4.1 — LinkedIn UGC Posts API// Post to LinkedIn on behalf of userawait fetch("https://api.linkedin.com/v2/ugcPosts", {  method: "POST",  headers: {    Authorization: `Bearer ${linkedinToken}`,    "Content-Type": "application/json",    "X-Restli-Protocol-Version": "2.0.0"  },  body: JSON.stringify({    author: `urn:li:person:${linkedinUrn}`,    lifecycleState: "PUBLISHED",    specificContent: {      "com.linkedin.ugc.ShareContent": {        shareCommentary: {          text: `Just solved "${title}" on LeadCode! 🚀\nDifficulty: ${difficulty} | Language: ${language}\nView solution: ${githubLink} #coding #${difficulty} #LeadCode`        },        shareMediaCategory: "NONE"      }    },    visibility: {      "com.linkedin.ugc.MemberNetworkVisibility": "PUBLIC"    }  })});Step 4.2 — User Controls for LinkedIn PostingAlways give users full control. Never auto-post without explicit consent:Global toggle in Settings: "Auto-post accepted solutions to LinkedIn" (default: OFF)Per-submission checkbox in the submit dialog: "Share this on LinkedIn"Daily digest mode: instead of one post per problem, batch into one end-of-day summaryCooldown setting: "Only post if I solve at least 3 problems today" — avoids single-problem postsSTEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION4.1a — UGC API DeprecatedLinkedIn is gradually deprecating the UGC Posts API in favor of Posts API (v2). Using the wrong endpoint can cause sudden failures.Check LinkedIn developer changelog at the time of implementation. As of 2025, ugcPosts still works but monitor for deprecation notices. Abstract the post function so swapping the endpoint requires changing one file.4.1b — Post DuplicateIf the background job retries due to a timeout, the same solution could get posted to LinkedIn twice.Before posting, write a pending_post record to DB with a unique key (user_id + problem_id + submission_id). Use a DB unique constraint. The job checks this before posting. Mark as posted=true after success.4.1c — Token Expired Mid-PostLinkedIn tokens expire at 60 days. If the access_token expires, the post silently fails.Wrap every LinkedIn API call in a try-catch. On 401, attempt token refresh using the refresh_token. If refresh also fails, set linkedin_connected = false and send user notification. Never swallow the error silently.Phase 5 — Settings & Connections UIThe Connections settings page is where users manage their GitHub and LinkedIn integrations. It needs to be clear, trustworthy, and give full control.Step 5.1 — Connections Page LayoutEach connected service should show: connection status, connected account name, what permissions were granted, action buttons to change or disconnect, and auto-push settings.GitHub Section:Status badge: "Connected as @username" (green) or "Not connected" (gray)Target repo selector: shows currently selected repo with a "Change" buttonCheckbox: "Auto-push to GitHub on accepted submissions" (default: ON once connected)Checkbox: "Auto-generate README per solution" (default: ON)"Disconnect" button with confirmation dialogLinkedIn Section:Status badge: "Connected as Full Name" or "Not connected"Checkbox: "Auto-post to LinkedIn on accepted submissions" (default: OFF)Dropdown: "Post frequency" — Immediately / Daily digest at 8 PM / Never"Disconnect" buttonSTEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION5.1 — Disconnect ConsequencesIf a user disconnects GitHub mid-session, pending background jobs may try to push using the now-invalid token.When user disconnects, immediately set github_connected = false and delete the stored token. Jobs check this flag before running. If false, skip the push and log: "Push skipped — GitHub disconnected."5.2 — Repo ChangeUser changes their target GitHub repo. Previous solutions were pushed to the old repo. New solutions go to the new repo. The history is split.When user changes repo, show a warning: "Previous solutions pushed to [old-repo] will not be migrated automatically. Only future solutions will go to [new-repo]." Optionally offer a "Migrate all solutions" button as a background job.Phase 6 — Background Job QueueGitHub pushes and LinkedIn posts must NOT block the submission response. They run asynchronously in a job queue so the user gets their verdict instantly.Step 6.1 — Why a Job Queue?Without a queue: the HTTP request waits for GitHub API + LinkedIn API before returning a response to the user. This adds 1-3 seconds of latency on every accepted submission, and if either API is down, the submission response fails entirely.With a queue: the submission returns "Accepted" in under 200ms. The GitHub and LinkedIn tasks run in the background within seconds. Failures can be retried without affecting the user experience.Step 6.2 — Inngest Setup (Recommended)Inngest is a serverless-friendly event queue that works natively with Next.js and Vercel. No Redis or RabbitMQ server needed.// Define the job (in inngest/functions.ts)export const onSubmissionAccepted = inngest.createFunction(  { id: "on-submission-accepted", retries: 3 },  { event: "submission/accepted" },  async ({ event, step }) => {    const { userId, problemId, code, language, submissionId } = event.data;    await step.run("push-to-github", async () => {      await pushToGitHub({ userId, problemId, code, language });    });    await step.run("post-to-linkedin", async () => {      await postToLinkedIn({ userId, problemId, submissionId });    });  });// Trigger from your API route after verdict:await inngest.send({  name: "submission/accepted",  data: { userId, problemId, code, language, submissionId }});STEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION6.1 — Job FailureGitHub API returns 500. The push fails. User never knows their solution was not saved.Inngest retries automatically (3x with exponential backoff). After all retries fail, write a failed_push record to DB and show an in-app notification: "GitHub push failed for [problem]. Retry now?"6.2 — Duplicate JobsNetwork timeout causes the job to be enqueued twice for the same submission. User gets two GitHub commits and two LinkedIn posts.Use submissionId as the Inngest event ID (idempotency key). Inngest deduplicates events with the same ID within a 24-hour window. This ensures exactly-once execution per submission.6.3 — Queue MonitoringBackground jobs are invisible. You have no idea if they are succeeding or silently failing at scale.Inngest provides a dashboard at app.inngest.com showing all events, function runs, and retry history. Set up alerting on failure rate > 5% using their webhook notifications to your Slack or Discord.Phase 7 — User Profile & Submission HistoryEvery user gets a public profile page showing their solved problems, streak, and a link to their GitHub solutions repo. This is their live coding portfolio.Step 7.1 — Profile Page FeaturesTotal problems solved, broken down by difficulty (Easy / Medium / Hard)Current solving streak (days) with a heatmap calendar (similar to GitHub contributions)Language distribution chart — how many problems solved in each languageLink to their GitHub solutions repo with a star countRecent accepted solutions list with problem name, difficulty, language, and timestampPublic shareable URL: leadcode.app/@usernameStep 7.2 — Submission History PageAll submissions — accepted and rejected — with full code and verdictFilter by: problem, language, verdict, date rangeClick any submission to see the code, test case results, and runtime/memory stats"Push to GitHub" manual button for accepted submissions where auto-push failedSTEPPROBLEM / CHALLENGEFIX & IMPLEMENTATION7.1 — Public Profile PrivacySome users do not want their solving activity publicly visible.Add a "Profile visibility" toggle in settings: Public / Private. Private profiles show only to the logged-in user. Default to Public since the GitHub integration already makes solutions public.7.2 — Streak CalculationIf a user is in UTC+5:30 and solves at 11:45 PM, the next solve at 12:15 AM (same calendar day for them) breaks the streak by UTC calculation.Store user timezone at signup. Calculate streaks in the user's local timezone, not UTC. Use the date-fns-tz library for reliable timezone-aware date arithmetic.8. Critical Security Reminders⚠  CRITICAL  Never store GitHub or LinkedIn tokens in plain text. Always encrypt at rest with AES-256-GCM.⚠  CRITICAL  Never expose your GitHub OAuth client_secret or LinkedIn client_secret in frontend code. Token exchange must happen server-side only.⚠  CRITICAL  Never send expected test case outputs to the browser. All verdict comparisons happen server-side.⚠  CRITICAL  Judge0 must run on a separate server from your main app. Never allow user code to run on the same machine as your database or API.✅  GOOD  Rate-limit all API endpoints. Max 10 code submissions per user per minute. Max 100 API requests per user per hour.✅  GOOD  Use Inngest idempotency keys to prevent duplicate GitHub commits or LinkedIn posts on job retries.LeadCode Implementation Plan — End of Document
